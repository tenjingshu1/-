# 通用领域本体库与知识图谱构建全流程方案

> 版本：1.0  
> 日期：2026-07-20  
> 适用范围：城市管理、公安治安、自然资源、不动产、应急管理、市场监管、生态环境、交通运输、住建、司法行政、财政采购、医疗卫生、教育、人社等领域。  
> 核心目标：将真实文件、业务规则和领域知识构建成可检索、可推理、可追溯、可评估、可持续更新的“本体库 + 知识图谱”。

---

## 一、方案结论

任何领域都可以复用同一条主链路：

```text
业务目标与能力问题
        ↓
真实文件和元数据治理
        ↓
领域本体设计（概念、属性、关系、约束）
        ↓
文档解析、OCR、分块和条文定位
        ↓
本体约束下的实体、事件、行为、规则抽取
        ↓
实体对齐、规范化、去重、冲突处理
        ↓
知识图谱实例化与来源证据链
        ↓
结构、语义、法理、业务、追溯质量检查
        ↓
JSON / JSON-LD / CSV / 图数据库 / HTML / API
        ↓
检索问答、合规校验、辅助决策和持续更新
```

最重要的原则是：

```text
本体库决定“抽什么、怎么分类、允许怎样关联、怎样判定正确”；
知识图谱承载“从哪些真实文件抽出了哪些具体事实、规则和实例”；
来源证据链证明“这个节点和关系为什么可信”。
```

本体库不是知识图谱的缩小版，知识图谱也不是法规文件目录。二者必须解耦，但必须通过稳定的类型、ID 和关系契约保持一致。

## 二、三个基本概念

### 2.1 本体库

本体库是领域知识的模式层，相当于知识系统的“数据标准、分类体系和规则说明书”。它主要回答：

- 领域中有哪些稳定概念？
- 概念如何分层和分类？
- 每类概念有哪些属性？
- 概念之间允许建立哪些关系？
- 哪些关系是必需、禁止或有条件成立的？
- 同义词、口语、旧称如何映射为标准术语？
- 一个抽取结果怎样才算符合领域规则？

例如，“城市道路”“行政主体”“处罚标准”是本体概念；“某市某路某次占道经营案件”是知识图谱实例。

### 2.2 知识图谱

知识图谱是实例层和事实层。它把文件、条文、对象、场所、主体、行为、事件、处罚、证据、程序、状态和规则组织为节点与关系。

知识图谱主要回答：

- 某类行为由哪个主体处理？
- 适用哪一部法规、哪一条条文？
- 处罚金额、期限和裁量条件是什么？
- 需要收集哪些证据？
- 必须经过哪些程序？
- 某事件发生在哪里、当前处于什么状态？
- 某条结论能否追溯到原始文件？

### 2.3 来源证据链

来源证据链连接知识事实与原始资料，至少应包含：

- 来源文件唯一 ID；
- 文件标准名称；
- 文件路径或远程文档 ID；
- 文号、发布机关、效力层级、有效状态；
- 条、款、项或章节定位；
- 原文片段或片段哈希；
- 页码、块 ID 或字符偏移；
- 抽取时间、抽取器版本和置信度；
- 人工审核状态。

没有来源证据链的业务事实，只能作为待审核候选，不应直接进入生产知识图谱。

## 三、总体架构

建议将系统分为六层：

| 层次 | 主要内容 | 典型输出 |
|---|---|---|
| 数据源层 | PDF、Word、网页、数据库、RAGFlow、元数据表格 | 原始文件、文件清单 |
| 文档治理层 | 去重、版本、效力、OCR、分块、条文定位 | 文档主表、chunk 表 |
| 本体模式层 | 概念域、类、属性、关系、约束、词表 | ontology.json、SHACL/规则 |
| 知识抽取层 | 实体、行为、事件、条文、处罚、证据、程序 | 抽取候选 JSONL |
| 图谱实例层 | 节点、边、规则实例、来源证据 | graph.json、JSON-LD、数据库 |
| 应用服务层 | 检索、问答、校验、推理、可视化、API | HTML、接口、智能体 |

推荐采用“本体驱动 + 规则抽取 + 大模型抽取 + 人工审核”的混合方式：

- 确定性强的内容使用解析器和规则；
- 语义复杂的内容使用大模型；
- 处罚金额、期限、主体和程序等高风险事实必须经过确定性校验；
- 低置信度、冲突或无来源结果进入人工审核队列。

## 四、项目启动阶段

### 4.1 明确业务边界

先写清楚“做什么”和“不做什么”。至少确定：

- 领域名称；
- 地域范围；
- 文件层级范围；
- 时间范围；
- 业务场景范围；
- 是否包含已失效文件；
- 是否包含内部制度、案例和业务数据；
- 图谱用于展示、问答、辅助执法、风险识别还是自动校验；
- 是否允许接入个人信息或敏感数据。

边界示例：

```yaml
domain: 应急管理
jurisdiction:
  - 国家级
  - 部级
  - 湖北省
  - 武汉市
document_status:
  include: [现行有效, 即将施行]
  exclude: [已废止, 失效]
business_scope:
  - 风险隐患排查
  - 应急预案
  - 事故报告
  - 指挥调度
  - 救援处置
excluded_scope:
  - 非应急领域一般性文件
  - 无正文且无法 OCR 的文件
```

### 4.2 定义能力问题

能力问题是图谱必须能够回答的问题，也是后续验收用例。建议先准备 30 至 100 条。

通用能力问题示例：

1. 某类事件涉及哪些管理对象？
2. 某类违法行为由哪个行政主体处理？
3. 行政相对人可能有哪些类型？
4. 需要经过哪些程序？
5. 需要哪些证据？
6. 处罚种类、金额、期限和适用条件是什么？
7. 哪些情形从轻、减轻、不予或从重处理？
8. 某一结论来源于哪部文件、哪一条款？
9. 新旧文件冲突时应适用哪一版本？
10. 某事件状态会触发什么行为或程序？

### 4.3 角色分工

| 角色 | 责任 |
|---|---|
| 领域专家 | 确定概念边界、关系语义和业务正确性 |
| 法务或政策专家 | 审核法源、效力、条文和处罚规则 |
| 数据工程师 | 文件治理、解析、抽取流水线、数据发布 |
| 本体工程师 | 概念域、类层级、关系和约束设计 |
| 算法工程师 | LLM 抽取、实体对齐、评估和问答 |
| 产品人员 | 能力问题、交互、检索和应用验收 |
| 审核人员 | 低置信度、冲突和高风险事实复核 |

### 4.4 首批交付目标

不要一开始追求“节点越多越好”。第一期应优先完成：

- 文件清单完整；
- 核心概念域稳定；
- 高频场景可闭环；
- 节点和关系可追溯；
- 没有无效边和重复节点；
- 能回答核心能力问题；
- 可持续增量更新。

## 五、数据源发现与文件治理

### 5.1 数据源类型

可使用的数据源包括：

- 本地文件目录；
- RAGFlow 等知识库；
- 法规数据库或政府网站；
- 元数据 Excel/CSV；
- 业务系统数据库；
- 标准术语表、权责清单、处罚事项清单；
- 案件、工单、台账和空间数据；
- 专家整理的规则表。

### 5.2 文件准入规则

每个文件必须经过准入检查：

1. 文件真实存在，可读取；
2. 文件与目标领域相关；
3. 文件名称可识别；
4. 文件层级和地域明确；
5. 效力状态明确或标记为待确认；
6. 正文可提取或可通过 OCR 获取；
7. 重复版本可以识别；
8. 来源路径和文件 ID 可保留；
9. 不违反数据权限和隐私要求。

### 5.3 文件主表

建议为每个文件建立统一元数据：

```json
{
  "documentId": "DOC-000001",
  "standardizedFileName": "L0_100000_无文号_中华人民共和国XX法_现行有效_2025-01-01.pdf",
  "fileFullName": "中华人民共和国XX法",
  "documentNumber": "无文号",
  "issuer": "全国人民代表大会常务委员会",
  "legalLevel": "国家法律",
  "administrativeLevel": "国家级",
  "jurisdictionCode": "100000",
  "status": "现行有效",
  "publishDate": "2024-01-01",
  "effectiveDate": "2025-01-01",
  "sourceType": "local|ragflow|web|database",
  "sourcePath": "...",
  "contentHash": "sha256:...",
  "parseStatus": "success|partial|ocr_required|failed",
  "domainRelevance": 0.98
}
```

### 5.4 文件去重

不能只按文件名去重。建议依次使用：

1. 文件二进制哈希；
2. 正文规范化哈希；
3. 文件全称 + 文号；
4. 文件全称 + 发布机关 + 日期；
5. 文本相似度；
6. 人工确认。

保留重复版本时，应建立：

```text
旧版本 --被修订为--> 新版本
旧版本 --被废止于--> 废止决定
地方实施文件 --依据--> 上位法
```

### 5.5 文件相关性筛选

建议采用三级筛选：

- 一级：文件名、目录、标签和元数据规则；
- 二级：正文关键词和领域分类模型；
- 三级：领域专家审核边界文件。

不要仅因文件中出现一个领域关键词就纳入，也不要因文件标题较宽泛就直接排除。应结合适用范围、职责主体和具体条文判断。

### 5.6 文件审计输出

至少输出：

- 全量文件清单；
- 纳入文件清单；
- 排除文件清单及原因；
- 重复文件组；
- 已失效文件清单；
- OCR 失败文件清单；
- 缺失正文文件清单；
- 地域、层级、状态分布报告。

## 六、本体设计方法

### 6.1 先设计概念域，再设计节点

概念域是稳定的知识分区。通用治理或执法领域可参考以下模板：

| 概念域 | 说明 |
|---|---|
| 法律法规体系 | 法律、法规、规章、规范性文件、标准、条文 |
| 业务场景 | 领域内可独立开展管理或处置的业务场景 |
| 管理对象 | 被管理、检查、保护、许可或处罚的对象 |
| 部件或设施 | 对象组成部分、设施设备、附属物 |
| 场所或空间区域 | 行为发生、规则适用或对象所在的空间 |
| 事件 | 动态发生并需要识别、流转和处置的事项 |
| 行为 | 行政行为、业务行为、处置行为、许可行为 |
| 违法或风险类型 | 违法行为、风险隐患、异常类型 |
| 主体 | 行政主体、责任主体、协同主体、管理主体 |
| 相对人 | 自然人、法人、组织、经营者、责任单位 |
| 程序 | 受理、核查、立案、调查、告知、决定、执行、归档 |
| 证据 | 笔录、照片、视频、检测报告、电子数据等 |
| 处罚或处置 | 警告、罚款、拘留、整改、查封、移送等 |
| 裁量规则 | 从轻、减轻、不予、从重及幅度规则 |
| 事件状态 | 待核查、处理中、整改中、已结案等 |
| 协同处置 | 跨部门、跨层级、跨系统协同 |
| 约束规则 | 必填、基数、条件、禁止、时限和一致性约束 |

不同领域可以删除不适用的概念域，也可以增加“风险源、应急资源、监测指标、登记权利、资产、合同、疾病、药品”等领域特有概念域。

### 6.2 静态对象与动态事件双主干

推荐将业务本体分为两条主干：

```text
静态主干：对象 → 部件 → 场所/空间 → 属性 → 规则
动态主干：事件 → 行为 → 主体/相对人 → 程序 → 证据 → 处罚/处置 → 状态
```

两条主干通过“发生于、作用于、涉及、需要证据、适用规则、由其处理”等关系连接。

这种方式可以避免把“道路”与“占道经营事件”混为同一类，也便于事件识别后自动匹配对象规则。

### 6.3 四层结构

推荐使用四层结构：

| 层级 | 含义 | 示例 |
|---|---|---|
| L1 | 概念域根节点 | 管理对象、事件、行政主体 |
| L2 | 领域主干类别 | 道路空间、环境污染事件 |
| L3 | 细分标准概念 | 人行道、施工噪声事件 |
| L4 | 实例或规则断言 | 某路段、某条文、某处罚规则实例 |

L1 至 L3 通常属于本体模式层；L4 通常属于知识图谱实例层。法规文件和条文也可独立作为实例节点。

### 6.4 概念节点字段

```json
{
  "id": "event-noise-construction",
  "name": "施工噪声事件",
  "code": "DOMAIN-EVENT-005-002",
  "level": 3,
  "domain": "事件",
  "definition": "施工活动产生噪声并可能影响周边生活环境的事件类型。",
  "parentId": "event-environmental-pollution",
  "childrenIds": [],
  "synonyms": ["工地噪声", "施工扰民"],
  "deprecatedTerms": [],
  "properties": {
    "category": "leaf",
    "reviewStatus": "approved"
  },
  "relations": [],
  "sourceLaw": null,
  "createdAt": "2026-07-20",
  "updatedAt": "2026-07-20"
}
```

### 6.5 定义编写规则

定义不能只写“用于分类和管理”。合格定义应包含：

- 上位类别；
- 核心特征；
- 与相邻概念的边界；
- 必要时说明包含和排除范围；
- 不直接混入具体处罚结论。

定义模板：

```text
X 是 Y 的下位概念，指具有 A、B 特征的对象/事件/行为；
包含 C，不包含 D；用于 E 场景中的识别、归类和规则匹配。
```

### 6.6 标准名称与同义词

每个概念只保留一个标准名称。口语、简称、旧称和地方称呼放入同义词表。

例如：

```json
{
  "canonical": "人行道",
  "synonyms": ["步道", "街沿", "人行道路"],
  "forbiddenAliases": ["路边"],
  "contextRules": [
    "门店门前的路边在有道路属性时映射为人行道",
    "无法判断道路属性时保留为待消歧空间"
  ]
}
```

### 6.7 本体关系设计

关系必须有明确方向、定义、适用域和值域。

| 关系 | 方向示例 | 含义 |
|---|---|---|
| `subClassOf` | 人行道 → 道路空间 | 下位概念属于上位概念 |
| `instanceOf` | 某路段 → 人行道 | 实例属于概念 |
| `appliesTo` | 法规 → 场景 | 法规适用于场景 |
| `regulatedBy` | 行为 → 法规/条文 | 行为受法规约束 |
| `occursIn` | 事件 → 空间 | 事件发生于空间 |
| `targets` | 行为 → 对象 | 行为作用于对象 |
| `handledBy` | 事件 → 主体 | 事件由主体处理 |
| `hasCounterparty` | 行为 → 相对人 | 行为面向相对人 |
| `requiresEvidence` | 违法/程序 → 证据 | 判定或程序需要证据 |
| `hasProcedure` | 行为/事件 → 程序 | 包含或需要程序 |
| `hasPenalty` | 违法 → 处罚 | 违法对应处罚 |
| `hasDiscretionRule` | 处罚 → 裁量规则 | 处罚受裁量规则约束 |
| `hasStatus` | 事件 → 状态 | 事件具有状态 |
| `requiresCollaboration` | 事件 → 协同处置 | 事件需要协同 |
| `basedOnClause` | 规则 → 条文 | 规则来源于条文 |
| `supersedes` | 新文件 → 旧文件 | 新文件替代旧文件 |

### 6.8 关系契约

为每种关系定义契约：

```json
{
  "type": "hasPenalty",
  "label": "对应处罚",
  "sourceDomains": ["违法类型"],
  "targetDomains": ["处罚类型", "处罚标准"],
  "cardinality": "0..n",
  "inverse": "penaltyFor",
  "transitive": false,
  "symmetric": false,
  "requiresProvenance": true,
  "minimumConfidence": 0.9
}
```

关系契约可以阻止错误关系，例如“处罚标准发生于空间区域”或“证据类型由相对人执法”。

### 6.9 属性设计

不要把所有信息都建成节点。判断规则：

- 需要单独检索、复用、关联或推理的信息，建节点；
- 仅描述一个节点且没有独立语义的信息，建属性；
- 具有条件、来源和适用范围的复杂值，建规则节点或关系实例。

适合做属性的内容：

- 名称、编码、日期、状态、数值、单位；
- 页码、条款号、来源路径；
- 置信度、审核状态、版本号；
- 经度、纬度、地址等实例属性。

### 6.10 约束设计

常见约束包括：

- 必填约束：文件实例必须有来源文件；
- 基数约束：处罚标准至少关联一条来源条文；
- 类型约束：`hasPenalty` 的源必须是违法类型；
- 唯一约束：标准文件名或节点 ID 唯一；
- 条件约束：只有满足特定情节才适用从轻处罚；
- 范围约束：金额必须包含币种和上下限；
- 时间约束：程序期限必须包含数值、单位和起算点；
- 禁止约束：失效法规不得作为当前有效结论的唯一依据；
- 一致性约束：关系的目标名称必须与目标节点名称一致。

## 七、文档解析与分块

### 7.1 解析顺序

推荐顺序：

1. 读取文件元数据；
2. 提取可复制文本；
3. 检测扫描件；
4. OCR；
5. 清理页眉、页脚、水印和乱码；
6. 识别标题、章、节、条、款、项；
7. 生成结构化条文；
8. 按条款和语义分块；
9. 保存页码和字符偏移；
10. 计算块哈希。

### 7.2 分块原则

法规类文件优先按“条”分块，不要固定每 500 字机械切割。一个 chunk 应尽量包含完整规则：

```text
主体 + 条件 + 行为 + 对象 + 处罚/处置 + 例外 + 条款号
```

超长条文可以按款或项切分，但必须保留父条文上下文。

### 7.3 chunk 数据结构

```json
{
  "chunkId": "DOC-000001-ARTICLE-32-P2",
  "documentId": "DOC-000001",
  "chapter": "第四章 法律责任",
  "article": "第三十二条",
  "paragraph": 2,
  "pageStart": 12,
  "pageEnd": 13,
  "text": "...",
  "contextBefore": "...",
  "contextAfter": "...",
  "contentHash": "sha256:..."
}
```

### 7.4 OCR 质量控制

OCR 后重点检查：

- 条款号是否识别错误；
- 数字金额是否缺位；
- “以上/以下/不超过”是否丢失；
- 日期和期限是否错误；
- 否定词是否丢失；
- 主体名称是否错字；
- 表格行列是否错位。

处罚金额、拘留期限、行政期限等数值应通过正则和人工抽样双重校验。

## 八、知识抽取方案

### 8.1 抽取不是“全文关键词生成节点”

一个文本片段只有满足以下条件之一才应建节点：

- 属于本体中的概念或实例类型；
- 能参与有效业务关系；
- 对检索、问答、校验或推理有价值；
- 有清晰来源和语义边界；
- 不是仅为凑数量的模板复制。

### 8.2 分阶段抽取

建议按以下顺序：

1. 文件实例；
2. 条文节点；
3. 主体和相对人；
4. 管理对象、场所、设施；
5. 行为、事件、违法或风险类型；
6. 处罚、处置和行政措施；
7. 证据和程序；
8. 金额、期限和裁量规则；
9. 状态与协同关系；
10. 复杂条件和约束规则。

先抽实体，再抽关系；先抽法定事实，再做归一化和推理。

### 8.3 规则抽取与大模型抽取分工

| 内容 | 推荐方法 |
|---|---|
| 文件名、文号、日期 | 规则和元数据解析 |
| 章、节、条、款、项 | 法规结构解析器 |
| 金额、比例、期限 | 正则 + 单位标准化 |
| 主体、行为、对象 | 词表 + NER + 大模型 |
| 复杂条件和例外 | 大模型 + 规则校验 |
| 条文间引用 | 正则 + 引用解析 |
| 同义词和概念映射 | 词典 + 向量召回 + 大模型判定 |
| 关系方向和类型 | 本体关系契约校验 |

### 8.4 大模型抽取输出要求

大模型必须输出严格 JSON，不允许输出未定义节点类型和关系类型。

```json
{
  "source": {
    "documentId": "DOC-000001",
    "chunkId": "DOC-000001-ARTICLE-32",
    "article": "第三十二条"
  },
  "nodes": [
    {
      "tempId": "n1",
      "type": "违法类型",
      "name": "未按规定登记",
      "definition": "...",
      "evidenceText": "...",
      "confidence": 0.96
    }
  ],
  "relations": [
    {
      "sourceTempId": "n1",
      "type": "hasPenalty",
      "targetTempId": "n2",
      "evidenceText": "...",
      "confidence": 0.94
    }
  ],
  "uncertainties": []
}
```

### 8.5 通用抽取提示词模板

```text
你是领域本体与知识图谱抽取器。

任务：从给定文件片段中抽取符合本体定义的节点、属性和关系。

必须遵守：
1. 只能使用给定的节点类型、关系类型和枚举值。
2. 不得补充文本中没有出现且无法由本体直接确定的事实。
3. 每个业务事实必须返回原文证据。
4. 金额、期限、比例必须保留上下限、单位、条件和适用对象。
5. 区分本体概念和具体实例。
6. 同一语义使用标准名称，同义词放入 aliases。
7. 无法确定时写入 uncertainties，不得猜测。
8. 输出必须符合 JSON Schema。

输入：
- 文档元数据：{{document_metadata}}
- 条文位置：{{article_location}}
- 本体节点类型：{{node_types}}
- 关系契约：{{relation_contracts}}
- 标准词表：{{terminology}}
- 文本：{{chunk_text}}
```

### 8.6 处罚标准细粒度抽取

处罚不能只抽“罚款”。应抽取：

- 处罚种类；
- 最低金额和最高金额；
- 金额单位；
- 最短期限和最长期限；
- 期限单位；
- 适用主体；
- 前置条件；
- 从轻、减轻、不予、从重情形；
- 是否责令改正；
- 是否没收违法所得；
- 是否存在并处、可以并处或选择处罚；
- 来源条文。

```json
{
  "type": "处罚标准",
  "penaltyType": "罚款",
  "amountMin": 1000,
  "amountMax": 5000,
  "currency": "CNY",
  "operator": "range",
  "condition": "拒不改正",
  "targetType": "单位",
  "sourceClauseId": "DOC-000001-ARTICLE-32-P2"
}
```

### 8.7 关系实例化

一个复杂关系应保留关系本身的属性：

```json
{
  "id": "REL-000001",
  "sourceId": "violation-001",
  "type": "hasPenalty",
  "targetId": "penalty-standard-001",
  "condition": "逾期未改正",
  "jurisdiction": "国家级",
  "validFrom": "2025-01-01",
  "validTo": null,
  "sourceDocumentId": "DOC-000001",
  "sourceClauseId": "DOC-000001-ARTICLE-32",
  "evidenceText": "...",
  "confidence": 0.97,
  "reviewStatus": "approved"
}
```

## 九、规范化、实体对齐与去重

### 9.1 规范化顺序

```text
原始提及
  → 空白和符号清理
  → 简繁和全半角统一
  → 单位和日期标准化
  → 同义词映射
  → 标准名称匹配
  → 上下文消歧
  → 实体合并或新建
```

### 9.2 节点唯一键

不同节点类型使用不同唯一键：

| 节点类型 | 推荐唯一键 |
|---|---|
| 本体概念 | `domain + canonicalName` |
| 法规文件 | `fileFullName + documentNumber + versionDate` |
| 条文 | `documentId + article + paragraph + item` |
| 业务实例 | `sourceSystem + sourceRecordId` |
| 处罚标准 | `clauseId + penaltyType + condition + targetType` |
| 规则实例 | `sourceClauseId + normalizedRuleHash` |
| 空间实例 | `administrativeCode + standardizedAddress + geometryHash` |

### 9.3 语义重复处理

以下情况通常应合并：

- 标点不同；
- “文件实例”后缀不同；
- 简称与全称；
- 同一文号的同一版本；
- 同义词映射到同一标准概念；
- 相同条文产生的同义规则。

以下情况不能简单合并：

- 同名但不同发布机关；
- 同名但不同地域；
- 同名但不同版本；
- 同一处罚名称但金额、条件不同；
- 同一对象名称但实际是不同空间实例。

### 9.4 冲突处理

冲突事实不要覆盖删除，应并存并标记：

```json
{
  "conflictGroupId": "CONFLICT-001",
  "status": "pending_review",
  "facts": ["FACT-001", "FACT-002"],
  "conflictType": "penalty_amount",
  "resolution": null
}
```

法理冲突处理应考虑：

- 上位法优于下位法；
- 特别法优于一般法；
- 新法优于旧法；
- 地方文件不得违反上位法；
- 适用地域和主体范围；
- 生效、修订、废止时间。

## 十、知识图谱构建

### 10.1 图谱节点组成

生产图谱通常包含：

1. 本体概念节点；
2. 法规文件节点；
3. 条文节点；
4. 业务对象和设施实例；
5. 场所和空间实例；
6. 主体和相对人实例；
7. 事件实例；
8. 行为实例；
9. 违法、风险或异常实例；
10. 处罚和处置标准；
11. 程序、证据、状态和协同节点；
12. 规则实例和关系实例；
13. 来源 chunk 和审核记录。

### 10.2 图谱边组成

每条边至少包含：

- 唯一 ID；
- 源节点 ID；
- 关系类型；
- 目标节点 ID；
- 来源；
- 置信度；
- 审核状态；
- 有效时间；
- 抽取版本。

### 10.3 图谱生成伪代码

```javascript
const nodeMap = new Map();
const edgeMap = new Map();

function addNode(node) {
  const key = buildCanonicalKey(node);
  const existing = nodeMap.get(key);
  if (existing) return mergeNode(existing, node);
  validateNodeAgainstOntology(node);
  nodeMap.set(key, node);
  return node;
}

function addEdge(edge) {
  validateRelationContract(edge, nodeMap);
  requireProvenance(edge);
  const key = `${edge.sourceId}|${edge.type}|${edge.targetId}|${edge.condition || ''}`;
  if (!edgeMap.has(key)) edgeMap.set(key, edge);
}
```

### 10.4 不允许的构建方式

- 为了达到上千节点而复制模板节点；
- 把编码、普通属性和文件路径全部做成节点；
- 给每个文件机械生成相同的对象、行为和证据节点；
- 没有原文依据却自动生成具体处罚金额；
- 只建立“文件属于法规层级”关系，不抽业务知识；
- 通过创建无意义根节点掩盖孤立节点；
- 用完全相同的名称创建多个实例；
- 将本体概念和业务实例混在同一层级。

## 十一、来源追溯设计

### 11.1 最低追溯要求

| 内容 | 最低来源要求 |
|---|---|
| 本体概念 | 专家定义或来源文件集合 |
| 法规文件实例 | 本地路径或远程文档 ID |
| 条文节点 | 文件 ID + 条款号 + 原文 |
| 违法/风险节点 | 至少一个条文或业务记录 |
| 处罚标准 | 必须关联具体条文和条件 |
| 关系实例 | 必须有证据文本或规则依据 |
| 推理结论 | 必须保留推理路径和前提事实 |

### 11.2 一跳与多跳追溯

理想追溯路径：

```text
问答结论
  → 图谱关系实例
  → 处罚/规则节点
  → 条文节点
  → 法规文件实例
  → 原始文件路径或 RAGFlow 文档 ID
```

### 11.3 问答返回格式

知识问答不能只返回自然语言答案，建议同时返回：

```json
{
  "answer": "...",
  "facts": ["FACT-001", "FACT-002"],
  "reasoningPath": [
    "违法行为 --对应处罚--> 处罚标准",
    "处罚标准 --来源条文--> 第三十二条"
  ],
  "citations": [
    {
      "documentName": "中华人民共和国XX法",
      "article": "第三十二条",
      "sourcePath": "...",
      "evidenceText": "..."
    }
  ],
  "confidence": 0.95
}
```

## 十二、质量检查体系

### 12.1 文件覆盖检查

```text
文件覆盖率 = 已进入图谱的有效文件数 / 应纳入文件数
正文可用率 = 正文可抽取文件数 / 纳入文件数
条文覆盖率 = 已生成条文节点文件数 / 有结构化条文文件数
```

建议门槛：

- 文件覆盖率 100%；
- 正文可用率不低于 95%；
- 高价值文件条文覆盖率 100%；
- 被排除文件必须有原因。

### 12.2 结构检查

必须自动检查：

- 节点 ID 是否唯一；
- 边 ID 是否唯一；
- 是否存在目标不存在的无效边；
- 父节点是否存在；
- 是否存在层级环；
- 节点类型是否在本体中定义；
- 关系类型是否符合域和值域；
- 必填属性是否完整；
- 文件实例是否有来源；
- 是否存在非法自环。

生产门槛：无效边、重复 ID、缺失父节点、非法关系均为 0。

### 12.3 重复检查

至少统计：

- 完全重复节点；
- 标准化名称重复节点；
- 高相似度候选重复节点；
- 重复关系；
- 重复文件实例；
- 同条文重复规则。

重复不是一律删除。必须先判断是同义、版本、地域、条件还是实例差异。

### 12.4 孤立节点检查

孤立节点定义为入度和出度均为 0 的节点。

允许孤立的情况应非常有限，例如：

- 待审核候选节点；
- 尚未完成抽取的临时节点；
- 明确标记的外部占位节点。

正式发布图谱孤立节点应为 0。更重要的是检查“弱连接节点”：只有一条无业务意义的层级关系，却没有来源或业务关系的节点也应列入审核。

### 12.5 跨概念域连接检查

每个核心概念域应与业务链路中的其他域连接。例如：

```text
事件 → 发生于 → 空间区域
事件 → 涉及 → 管理对象
事件 → 需要协同 → 协同处置
事件 → 具有状态 → 事件状态
违法 → 对应处罚 → 处罚标准
程序 → 需要证据 → 证据类型
```

概念域只有内部父子关系、没有跨域关系，通常说明本体尚未形成业务闭环。

### 12.6 语义正确性检查

采用抽样人工评审计算：

```text
节点准确率 = 正确节点数 / 抽样节点数
关系准确率 = 正确关系数 / 抽样关系数
属性准确率 = 正确属性数 / 抽样属性数
来源准确率 = 可由来源直接支持的事实数 / 抽样事实数
```

建议门槛：

- 一般节点准确率不低于 95%；
- 高风险处罚和程序关系准确率不低于 99%；
- 来源准确率 100%；
- 处罚金额、期限和适用条件应进行全量规则校验。

### 12.7 能力问题验收

对项目启动时定义的问题进行回归测试：

- 是否能找到答案；
- 路径是否正确；
- 是否引用正确文件和条文；
- 是否使用有效版本；
- 是否遗漏条件和例外；
- 是否能明确表示“不确定”或“没有依据”。

### 12.8 100 分评价模型

| 指标 | 分值 |
|---|---:|
| 文件来源与范围正确 | 10 |
| 文件覆盖与正文可用 | 10 |
| 本体概念域和层级完整 | 10 |
| 节点类型与命名一致 | 10 |
| 业务节点覆盖 | 10 |
| 关系正确性与业务闭环 | 15 |
| 处罚、程序、证据细粒度 | 10 |
| 来源追溯完整 | 15 |
| 去重、孤立、无效边 | 5 |
| 能力问题和问答验证 | 5 |

建议发布门槛：总分不低于 90，且来源追溯、无效边、处罚事实三个关键项不得失分。

## 十三、人工审核闭环

### 13.1 自动进入审核队列的情况

- 置信度低于阈值；
- 缺少来源条文；
- 金额或期限解析冲突；
- 同一实体出现多个标准名称；
- 新关系类型不在本体中；
- 文件效力状态不明确；
- 新旧版本结论冲突；
- 跨域关系不符合契约；
- 大模型输出无法通过 JSON Schema；
- 问答结果只有推测没有证据。

### 13.2 审核状态

```text
candidate → auto_validated → pending_review → approved/rejected → published
```

### 13.3 审核记录

审核记录必须包含审核人、时间、修改前后值、理由和来源，不能只覆盖原值。

## 十四、发布格式与应用

### 14.1 推荐交付格式

| 文件 | 用途 |
|---|---|
| `ontology.json` | 本体主数据 |
| `ontology.ts` | 前端项目直接引用 |
| `graph.json` | 通用图谱交换 |
| `graph.jsonld` | 语义网和 RDF 兼容 |
| `nodes.csv` | 批量导入和检查 |
| `edges.csv` | 批量导入和检查 |
| `documents.csv` | 文件覆盖审计 |
| `chunks.jsonl` | 条文与原文追溯 |
| `quality-report.json` | 自动质量报告 |
| `audit-report.md` | 人工验收报告 |
| `viewer.html` | 交互可视化 |

### 14.2 JSON 不是运行程序

JSON 是数据底座，不能直接运行。HTML、前端应用、后端服务或图数据库读取 JSON 后提供交互能力。

### 14.3 图数据库导入

图数据库中建议建立：

- 节点类型索引；
- 节点唯一键约束；
- 文件 ID 和条文 ID 索引；
- 标准名称全文索引；
- 空间索引；
- 有效时间索引。

### 14.4 可视化要求

可视化页面至少应具备：

- 搜索节点和关系；
- 按节点类型筛选；
- 按文件、地域、层级、状态筛选；
- 缩放、平移、重置；
- 点击节点查看属性和来源；
- 高亮一跳或多跳关系；
- 概念域收起与节点展开；
- 查看条文原文；
- 显示节点数、关系数和质量状态；
- 大图谱采用按需加载，不一次渲染全部节点。

### 14.5 RAG 与知识图谱融合

推荐执行顺序：

```text
用户问题
  → 本体识别问题涉及的概念类型
  → 图谱检索候选实体和关系路径
  → 根据来源条文 ID 检索 RAG chunk
  → 大模型基于图谱事实和原文生成答案
  → 返回答案、路径、文件、条文和置信度
```

图谱负责结构化定位和推理，RAG 负责提供原文上下文。不能仅凭向量相似度直接生成高风险结论。

## 十五、增量更新与版本治理

### 15.1 版本号

建议采用：

```text
ontology: 领域-主版本.次版本.修订版本
graph: 领域-YYYYMMDD-构建号
extractor: 抽取器版本 + 提示词版本 + 模型版本
```

### 15.2 增量更新流程

```text
发现新增或修改文件
  → 文件哈希和版本判断
  → 只解析变化文件
  → 重新抽取受影响条文
  → 实体对齐
  → 计算新增、修改和失效事实
  → 运行全量结构检查和局部语义检查
  → 人工审核
  → 发布新版本
```

### 15.3 失效处理

文件失效时不要物理删除历史事实，应：

- 标记文件状态；
- 设置事实有效截止时间；
- 建立被废止或被替代关系；
- 默认问答只使用当前有效事实；
- 历史查询仍可访问旧版本。

### 15.4 影响分析

更新文件后，应通过图谱反向关系查找：

- 受影响的处罚标准；
- 受影响的程序和证据要求；
- 受影响的业务场景；
- 受影响的问答模板；
- 需要重新审核的推理规则。

## 十八、自动验证代码骨架

```javascript
function validateGraph(graph, ontology) {
  const errors = [];
  const nodeById = new Map();
  const edgeKeys = new Set();

  for (const node of graph.nodes) {
    if (nodeById.has(node.id)) errors.push(`重复节点ID: ${node.id}`);
    nodeById.set(node.id, node);
    if (!ontology.nodeTypes[node.type]) errors.push(`未知节点类型: ${node.type}`);
  }

  for (const edge of graph.edges) {
    if (!nodeById.has(edge.source)) errors.push(`缺失源节点: ${edge.source}`);
    if (!nodeById.has(edge.target)) errors.push(`缺失目标节点: ${edge.target}`);

    const key = `${edge.source}|${edge.type}|${edge.target}|${edge.condition || ''}`;
    if (edgeKeys.has(key)) errors.push(`重复关系: ${key}`);
    edgeKeys.add(key);

    validateContract(edge, nodeById, ontology.relationContracts, errors);
    if (requiresProvenance(edge) && !edge.sourceClauseId) {
      errors.push(`关系缺少条文来源: ${edge.id}`);
    }
  }

  const degrees = new Map(graph.nodes.map(node => [node.id, 0]));
  for (const edge of graph.edges) {
    degrees.set(edge.source, (degrees.get(edge.source) || 0) + 1);
    degrees.set(edge.target, (degrees.get(edge.target) || 0) + 1);
  }
  for (const [id, degree] of degrees) {
    if (degree === 0) errors.push(`孤立节点: ${id}`);
  }

  return errors;
}
```

## 十九、常见失败模式与修复

### 19.1 节点很多但没有业务价值

表现：大量编码、类别属性、本地路径和模板字段被做成节点。

修复：重新应用“是否需要独立检索、复用、关联或推理”的判断；普通字段降级为属性。

### 19.2 每个文件只有一个法规节点

表现：图谱只是文件目录，缺少对象、行为、违法、处罚、程序和证据。

修复：按条文抽取业务事实，建立文件 → 条文 → 规则 → 业务概念的链路。

### 19.3 文件节点孤立

表现：文件实例只有名称，没有与法规标准、条文或业务节点连接。

修复：至少建立 `instanceOf`、`containsClause` 和业务事实的 `derivedFrom` 关系。

### 19.4 概念域卡片没有跨域关系

表现：域内父子关系完整，但事件状态、空间区域、协同处置等概念域与其他域断开。

修复：补充事件 → 空间、事件 → 状态、事件 → 协同、协同 → 主体、状态 → 程序等关系契约和实例关系。

### 19.5 语义重复节点

表现：“现场照片”“现场图片”“照片证据”分别成为节点。

修复：建立标准术语、同义词和实体对齐规则，只保留一个标准概念。

### 19.6 处罚事实不完整

表现：只抽到“罚款”，没有金额、期限、对象、条件和条款。

修复：使用处罚专用结构化抽取器，并对数字、单位、比较词和条件做确定性校验。

### 19.7 关系数量看似充足但意义很弱

表现：所有节点只连接根节点或文件节点。

修复：统计业务关系占比、跨域连接和能力问题路径，不用层级边掩盖业务关系不足。

### 19.8 大模型补写不存在的法规或条文

表现：图谱中出现本地或知识库中没有的文件。

修复：文件节点只能从文件主表创建；模型无权创建未知文件；所有来源 ID 必须通过白名单校验。

### 19.9 可视化一次加载上万节点

表现：黑屏、浏览器卡死、缩放困难。

修复：默认展示概念域或核心子图；搜索后按需展开；限制标签数量；使用分层布局、聚合边和局部加载。

## 二十二、一句话执行标准

一套合格的领域本体库和知识图谱，不以节点数量为核心，而以以下五点为核心：

1. 概念边界清楚；
2. 业务关系完整；
3. 每个事实真实可追溯；
4. 结构能够自动校验；
5. 能稳定回答真实业务问题。

最终应形成：

```text
真实文件是依据，
本体库是标准，
知识图谱是事实网络，
约束和质量检查是可信保障，
检索问答与规则校验是应用出口。
```
