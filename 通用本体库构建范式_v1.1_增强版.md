# 通用本体库构建范式（增强版）

> 版本：1.1  
> 日期：2026-07-24  
> 适用目标：面向任意业务领域，使用统一方法自动推导本体库，包括 L1 概念域、L2 主干概念、L3 细分概念、属性、关系域、关系契约、规则与约束。  
> 核心思想：这不是一份固定的领域概念清单，而是一套“领域无关的本体推导协议”。换领域时，不重写方法，只替换输入材料。

---

## 一、范式定位

通用本体库构建范式的目标，是让大模型在面对任意领域时，都按照同一套方法进行本体构建，而不是依赖人工为每个领域单独编写一套提示词。

它解决的问题是：

```text
输入任意领域材料
        ↓
识别业务目标和能力问题
        ↓
抽取候选业务术语
        ↓
映射到通用分析维度
        ↓
归纳 L1 概念域
        ↓
拆分 L2 主干概念
        ↓
拆分 L3 标准细分概念
        ↓
设计属性、关系域、关系契约、规则和约束
        ↓
校验候选本体
        ↓
输出可用于后续知识抽取和知识图谱构建的候选本体库
```

本范式强调：

```text
本体库不是文件目录；
本体库不是实例数据集合；
本体库不是简单概念树；
本体库是指导后续抽取、校验、入库和推理的语义标准。
```

---

## 二、核心原则

### 1. 领域无关，输入驱动

范式本身不预设某个领域一定有哪些概念域。

不要在固定提示词中写死：

```text
城市管理一定有城市管理对象、违法类型、处罚处置；
财政招采一定有采购人、供应商、采购项目；
出入管理一定有访客、门禁、通行权限。
```

而应该让模型根据输入材料推导：

```text
这个领域有哪些主体？
有哪些对象？
有哪些空间？
有哪些行为？
有哪些事件？
有哪些流程？
有哪些规则？
有哪些状态？
有哪些证据？
有哪些异常？
有哪些来源依据？
```

### 2. 能力问题驱动

本体库不是为了“列概念”，而是为了支撑后续业务问题。

每次构建本体库时，都要先定义能力问题。

能力问题示例：

```text
某个人是否有权限进入某区域？
某个采购项目采用了什么采购方式？
某类违法行为对应什么处罚？
某个合同是否完成验收？
某条结论来源于哪份文件、哪一条文？
```

能力问题的作用是反推：

```text
为了回答这些问题，需要哪些概念域？
需要哪些子概念？
需要哪些关系？
需要哪些规则和约束？
需要哪些来源证据？
```

### 3. 先抽候选术语，再归纳概念域

不要让模型直接生成 L1/L2/L3。

更稳的流程是：

```text
先从元数据、标题、字段、正文、表单、流程材料中抽候选术语；
再判断这些术语属于主体、对象、空间、事件、行为、流程、规则、状态、证据、风险中的哪一类；
再把同类术语聚合成 L1 概念域；
再在每个 L1 下拆出 L2 主干概念；
最后在 L2 下拆出 L3 标准细分概念。
```

### 4. 单一语义原则

一个概念节点应尽量只表达一个清晰、独立、稳定的业务语义。

不推荐：

```text
质疑和投诉
采购人和采购代理机构
检查和处罚
申请和审批
入园和出园
诊断和治疗
生产和质检
```

推荐拆成：

```text
质疑
投诉
采购人
采购代理机构
检查
处罚
申请
审批
入园
出园
诊断
治疗
生产
质检
```

但不能机械拆分固定术语。以下词虽然有复合结构，但在很多领域中可能是稳定术语：

```text
监督检查
履约验收
安全生产
招标投标
出入管理
质量控制
```

判断规则：

```text
如果 A 和 B 可以分别定义、分别有属性、分别参与关系、分别遵守规则，应拆分；
如果“A和B”是领域固定术语，可以作为概念域或复合概念保留；
如果保留复合概念，底层子概念仍应尽量拆细。
```

### 5. 概念域必须形成业务链路

如果一个本体库只有层级树，没有跨域关系，就只是分类表，不是可用本体。

每个核心概念域都要能回答：

```text
它和哪些概念域发生业务关系？
它参与哪些能力问题？
它能支撑哪些实体抽取、关系抽取、规则判断或来源追溯？
```

## 三、输入规范

本范式不限制原始材料格式。

原始输入材料可以包括：

```text
PDF 文件
Word 文档
Excel / CSV 表格
网页正文
数据库表
扫描件图片
系统日志
业务表单
RAGFlow 文档与 chunk
文件元数据
人工整理的能力问题
```

其中：

```text
PDF、Word、网页等非结构化材料，主要用于抽取业务概念、关系、规则、约束和来源证据；
Excel、CSV、数据库表等结构化材料，主要用于抽取字段、实例、属性和业务关系；
RAGFlow 等知识库，可用于获取文档元数据、chunk、摘要、向量召回片段和来源定位。
```

### 3.1 原始材料与标准化输入包的区别

```text
原始输入材料：PDF、Word、Excel、数据库、网页、日志等；
标准化输入包：经过解析、OCR、清洗、分块、条文定位后的 JSON / JSONL 数据。
```

也就是说：

```text
PDF 是原始材料；
JSON / JSONL 是工程化后的模型输入格式。
```

### 3.2 推荐标准化输入包

每个文件建议整理成如下结构：

```json
{
  "document_id": "DOC-000001",
  "file_name": "某某管理办法.pdf",
  "file_type": "pdf",
  "source_path": "...",
  "metadata": {
    "title": "某某管理办法",
    "issuer": "某某部门",
    "legal_level": "地方政府规章",
    "publish_date": "2026-07-20",
    "effective_date": "2026-08-01",
    "validity_status": "现行有效",
    "business_category": "城市管理",
    "reference_relations": [],
    "effect_relations": []
  },
  "representative_chunks": [
    {
      "chunk_id": "DOC-000001-C001",
      "reason": "总则/适用范围",
      "text": "第一条 为了规范……",
      "page": 1,
      "section": "第一条"
    }
  ],
  "fields": [],
  "competency_questions": []
}
```

### 3.3 代表性正文片段选择规则

不要只固定取前三个 chunk。前三个 chunk 通常只包含立法目的、适用范围、基本原则，容易遗漏程序、证据、处罚、状态、协同和裁量规则。

建议每个文件抽取：

```text
1. 文件标题和元数据；
2. 目录 / 章标题 / 节标题；
3. 前 3 个 chunk；
4. 每个主要章节的首个 chunk；
5. 含有高价值关键词的 chunk；
6. 涉及职责、对象、程序、监督、处罚、证据、时限、状态、协同的 chunk。
```

高价值关键词示例：

```text
负责、主管、监督、检查、审批、许可、备案、申请、受理、核查、处罚、罚款、责令、整改、限期、证据、材料、程序、期限、投诉、质疑、协同、移送、废止、修订、施行
```

## 四、三轴通用分类模式

任何领域本体都应从三条线同时推导：

```text
静态主干
动态主干
约束维度
```

这三条线不是最终概念域，而是推导概念域的通用分类模式。

### 4.1 静态主干

静态主干回答：

```text
这个领域里有哪些相对稳定的对象？
```

通常包括：

```text
主体
对象
资源
空间
设施
文件
组织
产品
资产
人员
```

示例：

```text
行政领域：行政主体、相对人、管理对象、空间区域
财政领域：预算项目、采购项目、供应商、合同、资金
医疗领域：患者、疾病、药品、检查、医疗机构
教育领域：学校、学生、教师、课程、学籍
工业领域：设备、产线、物料、工艺、产品
```

### 4.2 动态主干

动态主干回答：

```text
这个领域中会发生什么事？按什么过程流转？
```

通常包括：

```text
事件
行为
流程
程序
申请
审批
交易
诊疗
教学
生产
维修
处置
状态流转
```

示例：

```text
行政领域：受理、立案、调查、处罚、执行
财政领域：预算编制、采购、评审、合同、验收、支付
医疗领域：挂号、诊断、检查、治疗、用药、随访
教育领域：报名、选课、考试、评价、毕业
工业领域：生产、检测、报修、维护、质检、入库
```

### 4.3 约束维度

约束维度回答：

```text
什么是合法的？什么是禁止的？什么条件下成立？
```

通常包括：

```text
类型约束
关系方向约束
必填约束
基数约束
时限约束
状态流转约束
取值范围约束
权限约束
来源证据约束
冲突处理规则
```

示例：

```text
“供应商 — 提交 — 投标文件”方向合法，反过来不合法；
“处方”必须关联医生、患者、药品和时间；
“设备维修记录”必须关联设备、故障类型、处理人和处理时间；
“通行事件”必须关联主体、时间、点位和来源记录。
```

### 4.4 三轴模式使用原则

```text
静态主干用于发现对象类概念；
动态主干用于发现事件、行为、流程和状态；
约束维度用于形成关系契约、校验规则和业务判断条件。
```

不要只设计静态对象，也不要只设计流程事件。高质量本体需要三者同时存在。

## 五、通用分析维度

通用分析维度不是最终概念域，而是引导模型归纳概念域的思考框架。

模型面对任意领域时，都应先按以下维度分析输入材料：

| 分析维度 | 需要回答的问题 | 可能形成的概念域 |
|---|---|---|
| 来源依据 | 规则、事实、结论来自哪里？ | 文件、条文、制度、记录、来源证据 |
| 主体角色 | 谁负责、谁参与、谁被管理？ | 管理主体、参与主体、责任主体、相对人 |
| 管理对象 | 被管理、被操作、被保护、被服务的对象是什么？ | 业务对象、资源对象、管理对象 |
| 空间场景 | 事情发生在哪里？规则适用于哪里？ | 区域、场所、空间单元、网格 |
| 行为动作 | 谁做了什么动作？ | 行为、操作、审批、核验、处置 |
| 事件事项 | 发生了什么需要识别或处置的事项？ | 事件、事项、案件、申请、交易 |
| 流程程序 | 事情如何流转？有哪些步骤？ | 流程、程序、环节、阶段 |
| 规则约束 | 什么情况下允许、禁止、触发、失效？ | 规则、条件、约束、准入要求 |
| 状态结果 | 当前处于什么状态？产生什么结果？ | 状态、结果、结论、处理结果 |
| 证据记录 | 用什么证明事实发生过？ | 记录、日志、凭证、附件、证据 |
| 风险异常 | 什么情况是异常、风险、违规或失败？ | 风险、异常、违规类型、失败原因 |
| 协同责任 | 多个主体之间如何分工、协作、移交？ | 协同、责任、分工、移送、联动 |

注意：

```text
通用分析维度不是固定 L1；
它只是帮助模型从材料中推导该领域自己的 L1 概念域。
```

## 六、L1 / L2 / L3 归属判断标准

### 6.1 L1：概念域

L1 是最高层业务语义分区。

判断标准：

```text
1. 是否代表一类稳定业务维度；
2. 是否能支撑至少一个能力问题；
3. 是否能容纳一组稳定子概念；
4. 是否不是具体文件名、表名或字段名；
5. 是否能和其他概念域建立业务关系；
6. 是否对后续抽取、校验、推理有价值；
7. 是否有输入材料中的证据支持。
```

示例：

```text
行政领域：主体、相对人、程序、证据、处罚
财政领域：预算、采购、合同、验收、支付
医疗领域：患者、疾病、检查、诊断、治疗、药品
教育领域：学生、教师、课程、考试、学籍
工业领域：设备、工艺、物料、生产、质检
```

不应作为 L1 的内容：

```text
办法
通知
规定
2024年文件
数据库字段
项目编号
文件目录名称
临时标签
```

这些更像文件类型、时间标签、字段或目录，不是业务语义域。

### 6.2 L2：主干子概念

L2 是 L1 下的稳定类别。

判断标准：

```text
1. 能否通过“A 是 L1 中的一类”测试；
2. 是否稳定、可定义、可复用；
3. 是否不是具体实例；
4. 是否不是普通属性；
5. 同一 L1 下是否同维度、同粒度；
6. 是否能参与关系、规则或实例抽取；
7. 是否有来源证据。
```

示例：

```text
L1：采购主体
L2：采购人、采购代理机构、供应商、评审专家、财政部门

L1：医疗行为
L2：挂号、问诊、检查、诊断、治疗、用药、随访

L1：工业设备
L2：生产设备、检测设备、运输设备、仓储设备
```

不应作为 L2 的内容：

```text
张三
武汉市某医院
项目编号A001
合同金额
2026-07-20
状态=已完成
鄂A12345
```

这些是实例或属性值。

### 6.3 L3：标准细分概念

L3 是可以直接用于实例分类、三元组抽取、规则匹配的细分概念。

判断标准：

```text
1. 是 L2 的下位类型；
2. 具有明确边界；
3. 能作为实例的类型标签；
4. 能被正文、字段或业务记录稳定识别；
5. 可以承接属性、关系或规则；
6. 粒度不能细到具体实例；
7. 有来源证据或稳定业务词表支持。
```

示例：

```text
L1：采购文件
  L2：采购公告
    L3：招标公告、竞争性磋商公告、询价公告

L1：医疗检查
  L2：影像检查
    L3：CT检查、MRI检查、X光检查

L1：工业设备
  L2：生产设备
    L3：数控机床、注塑机、焊接机器人
```

L3 以下通常就是实例层。

例如：

```text
“CT检查” 是 L3 概念；
“患者张三于2026年7月20日做的CT检查记录” 是实例。
```

### 6.4 L1 / L2 / L3 的常见错误

```text
把文件类型当成 L1；
把字段名当成 L2；
把具体实体当成 L3；
同一 L1 下 L2 维度混乱；
L2 与 L3 粒度倒置；
将状态值、金额、编号误当成概念；
将固定术语机械拆分；
将并列概念强行保留为一个子概念。
```

## 七、自动化构建流程

### 阶段 1：领域画像生成

模型先根据输入材料生成领域画像。

需要识别：

```text
这个领域主要处理什么问题；
核心业务对象是什么；
核心参与主体是谁；
核心流程是什么；
核心规则是什么；
哪些内容不属于当前范围。
```

输出示例：

```json
{
  "domain_profile": {
    "domain_name": "东湖高新出入管理",
    "business_goal": "支撑人员、车辆、访客、权限、审批、通行事件和异常处置的本体建模",
    "core_tasks": ["权限判断", "访客审批", "通行核验", "异常处置", "记录追溯"],
    "boundary": "园区、楼栋、门岗、人员、车辆、通行权限及其审批处置过程",
    "excluded_scope": ["园区招商管理", "企业经营数据", "非出入相关人事管理"]
  }
}
```

### 阶段 2：候选术语抽取

模型从输入材料中抽取候选业务词，不直接确定本体。

候选术语应标注类型：

```text
actor：主体角色
object：管理对象
space：空间场景
behavior：行为动作
event：事件事项
process：流程程序
rule：规则约束
state：状态结果
evidence：证据记录
risk：风险异常
source：来源依据
other：其他
```

输出示例：

```json
{
  "candidate_terms": [
    {
      "term": "访客",
      "term_type": "actor",
      "evidence": "访客申请表字段：访客姓名、被访人、访问事由",
      "confidence": "high"
    },
    {
      "term": "门禁日志",
      "term_type": "evidence",
      "evidence": "门禁系统字段：通行时间、门岗、通行结果",
      "confidence": "high"
    }
  ]
}
```

### 阶段 3：候选术语归类与聚合

模型将候选术语映射到通用分析维度，并聚合出候选 L1 概念域。

聚合逻辑：

```text
同一类业务词数量较多；
能支撑至少一个能力问题；
能和其他概念域发生业务关系；
不是单纯文件类型；
不是普通字段集合；
具有稳定业务语义。
```

例如：

```text
访客、员工、施工人员、物业人员 → 出入主体
园区、楼栋、门岗、出入口、停车场 → 管理区域
门禁卡、二维码、人脸信息、车牌号 → 通行凭证
门禁日志、抓拍图片、审批记录 → 记录证据
```

### 阶段 4：批次抽取与预合并

如果输入文件较多，应分批发送给 LLM。

每批输出都必须保留：

```text
来源文件；
证据 chunk；
支持次数；
置信度；
候选 L1；
候选 L2；
候选 L3；
候选关系域。
```

分批结果不能直接作为正式本体，必须先做预合并。

预合并步骤：

```text
1. 名称标准化；
2. 精确去重；
3. 识别 “A和B” 形式候选复合概念；
4. 标记低置信度候选项；
5. 保留所有来源证据。
```

### 阶段 5：语义合并

不建议一次性把所有预合并结果直接交给 LLM。

更稳的流程是：

```text
预合并结果
  ↓
名称相似度 / 向量相似度聚类
  ↓
每个聚类交给 LLM 判断是否合并
  ↓
输出 old_name → canonical_name 映射表
  ↓
再汇总生成候选 L1/L2/L3 taxonomy
```

语义合并输出示例：

```json
{
  "canonical_name": "管理主体",
  "merged_terms": ["行政主体", "执法主体", "主管部门"],
  "not_merged_terms": ["行政相对人", "责任主体"],
  "reason": "前三者表示承担管理或执法职责的一方；后两者表示被管理或承担责任的一方。",
  "confidence": "high"
}
```

### 阶段 6：生成 L1 / L2 / L3 taxonomy

最终 taxonomy 应包含：

```text
L1 概念域；
L2 主干概念；
L3 标准细分概念；
概念定义；
归属理由；
来源证据；
置信度；
排除项；
同义词；
不应合并项。
```

### 阶段 7：属性识别

属性不是 L2 或 L3 子概念，除非它本身需要单独参与关系、检索、复用或推理。

属性识别原则：

```text
只描述某个对象，且没有独立业务语义 → 属性；
需要单独检索、关联、复用或推理 → 节点；
具有条件、来源和适用范围的复杂值 → 规则节点或关系实例。
```

输出字段：

```text
所属概念；
属性名称；
属性类型；
是否必填；
枚举值；
单位或格式；
来源证据。
```

### 阶段 8：关系矩阵生成

仅有 L1/L2/L3 不够，还必须生成跨域关系矩阵。

关系矩阵回答：

```text
哪个概念域连接哪个概念域；
通过什么关系连接；
方向是什么；
业务含义是什么；
是否有来源证据。
```

示例：

```json
{
  "relation_matrix": [
    {
      "source_domain": "出入主体",
      "relation": "提交",
      "target_domain": "出入申请",
      "direction": "出入主体 -> 出入申请",
      "business_meaning": "人员或车辆相关主体提交入园、访客或通行申请",
      "evidence": "访客申请表、车辆入园申请字段",
      "confidence": "high"
    }
  ]
}
```

### 阶段 9：关系契约生成

关系契约用于后续校验三元组是否合法。

每条关系契约至少包含：

```text
关系名称；
源类型；
目标类型；
方向；
基数；
是否需要来源证据；
是否可逆；
是否可传递；
非法示例。
```

示例：

```json
{
  "relation_contracts": [
    {
      "relation": "提交",
      "source_types": ["员工", "访客", "施工人员", "企业人员"],
      "target_types": ["访客申请", "车辆入园申请", "临时通行申请"],
      "direction": "出入主体 -> 出入申请",
      "cardinality": "0..n",
      "requires_evidence": true,
      "violation_example": "访客申请 — 提交 — 访客"
    }
  ]
}
```

### 阶段 10：规则与约束生成

规则用于业务判断，约束用于校验本体和抽取结果。

应生成：

```text
类型约束；
方向约束；
必填约束；
基数约束；
枚举约束；
时间约束；
状态流转约束；
禁止约束；
来源证据约束；
一致性约束。
```

示例：

```text
通行权限必须关联至少一个出入主体和一个管理区域；
门禁日志必须包含通行时间、通行点位、通行结果；
黑名单主体不得自动获得通行权限；
通行事件必须有来源记录或设备日志。
```

### 阶段 11：本体校验

候选本体必须经过自动校验。

校验项包括：

```text
1. L1 是否支撑能力问题；
2. L2 是否真的是 L1 的下位概念；
3. L3 是否真的是 L2 的下位概念；
4. 是否把实例误当成概念；
5. 是否把属性误当成概念；
6. 同一 L1 下的 L2 是否同层同维；
7. 同一 L2 下的 L3 是否同层同维；
8. 是否存在重复或同义概念未归一；
9. 是否存在不同概念被误合并；
10. 概念域之间是否形成业务链路；
11. 每个概念和关系是否有来源证据；
12. 关系方向是否明确；
13. 关系契约是否能拦截错误三元组；
14. 是否存在只有分类价值、没有业务推理价值的概念域；
15. 是否存在来源依据不足但置信度过高的问题；
16. 是否存在 “A和B” 未拆分或固定术语被误拆分的问题。
```

## 八、复合概念拆分规则

### 8.1 应拆分的情况

当 A 和 B 可以分别定义、分别具有属性、分别参与关系、分别受规则约束时，应拆分。

示例：

```text
质疑和投诉 → 质疑；投诉
采购人和采购代理机构 → 采购人；采购代理机构
招标文件和投标文件 → 招标文件；投标文件
中标和成交结果 → 中标结果；成交结果
申请和审批 → 申请；审批
入园和出园 → 入园；出园
```

### 8.2 不应机械拆分的情况

以下词可能是固定业务术语，不能仅凭字面拆分：

```text
监督检查
履约验收
招标投标
安全生产
出入管理
质量控制
应急管理
行政处罚
```

处理方式：

```text
如果它是固定术语，可作为概念域或复合业务概念保留；
如果内部包含可独立建模的子过程，应在下一层拆分；
如果原文中 A、B 分别有不同规则和关系，应拆分。
```

### 8.3 拆分判断提示

```text
1. A 和 B 是否能分别定义？
2. A 和 B 是否有不同主体？
3. A 和 B 是否有不同对象？
4. A 和 B 是否有不同流程？
5. A 和 B 是否有不同规则或时限？
6. A 和 B 是否能分别参与关系？
```

如果多数答案为“是”，应拆分。

## 九、不同领域的差异化适配要点

范式保持通用，但不同领域的本体重心不同。

| 领域 | 核心静态主干 | 核心动态主干 | 核心约束维度 | 特别注意 |
|---|---|---|---|---|
| 行政 / 执法 | 行政主体、相对人、管理对象、法规条文 | 受理、调查、处罚、执行、移送 | 法源效力、程序合法性、证据要求、裁量规则 | 来源条文和程序链必须可追溯 |
| 财政 / 招采 | 预算项目、采购人、供应商、采购项目、合同、资金 | 预算、采购、评审、合同、验收、支付 | 采购方式、资格条件、金额、期限、支付条件 | 资金流、合同流、采购流程要打通 |
| 医疗 | 患者、疾病、症状、检查、药品、机构 | 挂号、问诊、诊断、治疗、用药、随访 | 诊疗路径、禁忌、剂量、适应症、隐私权限 | 时间顺序、因果关系、风险提示很重要 |
| 教育 | 学校、学生、教师、课程、班级、学籍 | 招生、选课、授课、考试、评价、毕业 | 学籍规则、考试规则、课程修读条件 | 人员角色和生命周期管理很重要 |
| 工业 | 设备、物料、工艺、产线、产品、仓库 | 生产、质检、维修、入库、出库、报废 | 工艺参数、质量标准、安全规则、设备状态 | 设备状态、传感数据、故障事件要建好 |

换领域时，不能照搬某个领域的概念域清单。应只复用范式，重新从输入材料中推导该领域的 L1/L2/L3。

## 十一、通用提示词模板

```text
你是领域本体库构建器。

你的任务不是为某个固定领域背诵概念域，而是根据输入材料，按照通用本体库构建范式，自动推导该领域的 L1 概念域、L2 主干概念、L3 细分概念、属性、关系矩阵、关系契约、规则和约束。

一、请遵循三轴通用分类模式：
1. 静态主干：主体、对象、资源、空间、设施、文件、组织、人员、资产等相对稳定对象；
2. 动态主干：事件、行为、流程、申请、审批、交易、处置、状态流转等动态过程；
3. 约束维度：类型、方向、必填、基数、时限、状态流转、取值范围、权限、来源证据、冲突处理等规则。

二、请遵循以下通用分析维度：
1. 来源依据
2. 主体角色
3. 管理对象
4. 空间场景
5. 行为动作
6. 事件事项
7. 流程程序
8. 规则约束
9. 状态结果
10. 证据记录
11. 风险异常
12. 协同责任

这些分析维度不是最终概念域，只是帮助你归纳领域本体的思考框架。最终 L1 概念域必须从输入材料、能力问题和候选术语中归纳出来。

三、请按以下步骤执行：
1. 生成领域画像；
2. 抽取候选业务术语；
3. 将候选术语映射到通用分析维度；
4. 归纳 L1 概念域；
5. 拆分 L2 主干概念；
6. 拆分 L3 标准细分概念；
7. 识别属性；
8. 生成跨域关系矩阵；
9. 生成关系契约；
10. 生成规则和约束；
11. 输出质量问题和修改建议。

四、请严格遵守以下原则：
1. 不要把具体人员、具体机构、具体项目、具体文件、具体记录抽成概念；
2. 不要把编号、日期、金额、ID、经纬度等普通字段抽成概念；
3. L2 必须能通过“A 是 L1 中的一类”的测试；
4. L3 必须能通过“A 是 L2 的一种”的测试；
5. 同一 L1 下的 L2 要同层同维、粒度一致；
6. 同一 L2 下的 L3 要同层同维、粒度一致；
7. 每个 L1 至少要支撑一个能力问题；
8. 每个核心 L1 应能和其他 L1 建立业务关系；
9. 每个概念、关系、属性、规则、约束都必须有来源依据；
10. 不确定内容标记为 low，不要强行确认；
11. 遵循单一语义原则，避免将“A和B”式并列概念直接作为子概念；
12. 对固定术语不要机械拆分；
13. 不要输出解释性散文，只输出符合 JSON Schema 的结果。

五、输入材料：

六、输出格式：
请严格按照约定 JSON Schema 输出。
```

## 十三、最终判断标准

一套合格的通用本体构建范式，应满足：

```text
1. 换领域时不需要重写提示词主体；
2. 能从输入材料中自动抽取候选业务术语；
3. 能把业务术语归纳为合理的 L1 概念域；
4. 能把 L1 拆成稳定、同层同维的 L2 主干概念；
5. 能把 L2 拆成可用于实例分类和规则匹配的 L3 细分概念；
6. 能识别属性而不是把所有字段都建成概念；
7. 能遵循单一语义原则，处理“A和B”形式的复合概念；
8. 能生成跨域关系矩阵；
9. 能生成可校验的关系契约；
10. 能输出规则和约束；
11. 能标记不确定项；
12. 能为每个概念、关系、属性、规则、约束提供来源证据；
13. 能针对行政、财政、医疗、教育、工业等不同领域自动调整重点，而不复制固定领域清单。
```

最核心的一句话：

```text
通用本体库构建范式不是一张通用概念清单，
而是一套让模型从任意领域材料中自动推导本体结构的协议。
```
